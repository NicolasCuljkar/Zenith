🔐 6. Authentification & Sécurité

🔑
JWT HS256
Signé avec JWT_SECRET
Durée : 24h (configurable)
🔒
bcrypt (10 rounds)
Hashage des mots de passe
Mdp minimum 8 caractères
🛡️
Rate Limiting
Login : 10 req/min
Admin : 5 req/15min

Flux d'authentification

1.  Client   → POST /api/auth/login { email, password }
2.  Server   → bcrypt.compareSync(password, hash)
3.  Server   → jwt.sign({ id, name, email, role, is_admin }, JWT_SECRET, { expiresIn: '24h' })
4.  Server   ← { token, user }
5.  Client   → localStorage.setItem('zenith_token', token)
6.  Client   → Chaque requête : Authorization: Bearer <token>
7.  Middleware → jwt.verify(token, JWT_SECRET) → req.user
8.  Middleware → SELECT id FROM users WHERE id = ? (vérif. compte toujours existant)
9.  Réponse 401 → window.dispatchEvent(new CustomEvent('zenith:unauthorized'))
           → Retour automatique à l'écran de connexion

Middleware

MiddlewareComportementUsage
requireAuthVérifie JWT, retourne 401 si absent/invalide/expiré. Vérifie que l'user existe en base.Toutes les routes protégées
optionalAuthTente de décoder le JWT mais n'échoue pas si absent.Routes avec auth optionnelle (SSE)
requireAdminS'utilise après requireAuth. Vérifie req.user.is_admin === 1. Retourne 403 sinon.Routes /api/admin/*
errorHandlerGestionnaire global. Lit err.statusCode. Retourne { success: false, error }Fin du pipeline Express
⚠️
Compte admin par défaut (migration v6) Un compte admin est créé automatiquement : login admin, mot de passe admin. À changer immédiatement en production via POST /api/admin/users/:id/password ou directement en base.

0 13 1 * *

1er du mois à 13h — demande à chaque utilisateur de saisir son épargne du mois.

0 13 * * 1
Résumé budgétaire hebdomadaire

Chaque lundi — revenus nets, dépenses et épargne du mois en cours. Ajoute un avertissement si dépenses > 80% des revenus.

0 13 * * *
Vérification des alertes budgétaires

Chaque jour — évalue les 7 conditions d'alerte ci-dessous. Anti-spam : une alerte par clé par jour (stocké dans config).

⚡ 8. Frontend — Structure

Le frontend est une PWA mono-fichier : frontend/index.html (~8700 lignes). HTML, CSS et JS sont tous inline. Aucun bundler, aucun framework.

  • Totaux par catégorie + solde global en bas

Épargne (renderSavings)

  • Tableau mensuel par membre (une ligne = un mois)
  • Saisie/modification inline du solde total
  • Delta (variation mensuelle) calculé automatiquement côté backend
  • Graphique d'évolution du patrimoine

Analyse (renderAnalysis)

  • Filtre automatique : uniquement les mois avec revNet > 0 AND dep > 0
  • 4 métriques : revenu net moyen / dépenses / épargne / solde
  • Insights intégrés sous les métriques : taux épargne + projection, taux fixes, mois positifs
  • Score santé financière /100 : épargne (40 pts) + rigidité (30 pts) + niveau dépenses (30 pts)

  • Synchronisation des transactions
  • Catégorisation manuelle + import vers le suivi mensuel

🧮 10. Formules de calcul

Budget prévisionnel (entries)

// Revenus
revenu = Σ entries[cat='revenu'] impot = Σ entries[cat='impot'] revNet = revenu - impot
// Dépenses
fixe = Σ entries[cat='fixe'] variable= Σ entries[cat='variable'] loisir = Σ entries[cat='loisir'] dep = fixe + variable + loisir
// Indicateurs clés
epargne = Σ entries[cat='epargne'] rav = revNet - fixe - variable solde = revNet - dep - epargne
// Taux
depRate = dep / revNet × 100 fixRate = fixe / revNet × 100 epRate = epargne / revNet × 100 ravRate = rav / revNet × 100

Suivi mensuel réel (monthly_expenses + savings)

// Revenus réels
revA = overrides || budget prévu impA = overrides || budget prévu revNetA = revA - impA
// Dépenses réelles
fixA = base prévisionnelle + overrides varA = Σ monthly_expenses[variable] loiA = Σ monthly_expenses[loisir] // is_exceptional = 1 → exclus
// Épargne réelle
epDelta = savings.delta // du mois // delta = amount(N) - amount(N-1) // null si premier enregistrement // recalculé après update/delete
// Solde mensuel réel
depA = fixA + varA + loiA soldeA = revNetA - depA - max(0, epDelta)

Score santé financière (0-100)

// 30 pts — Rigidité des charges fixes
scoreRig = fixRate <= 30 ? 30
         : fixRate <= 50 ? round(30 - (fixRate-30) × 0.75)
         : 0

// 30 pts — Niveau des dépenses totales
scoreDep = depRate <= 50 ? 30
         : depRate <= 70 ? round(30 - (depRate-50) × 1.5)
         : 0

// 40 pts — Épargne / reste à vivre
scoreSav = min(40, ravRate × 1.2)

healthScore = scoreSav + scoreRig + scoreDep   // sur 100

🚀 12. Déploiement

Local (développement)

# 1. Cloner et installer
git clone <repo> && cd zenith/backend
npm install

# 2. Configurer l'environnement
cp .env.example .env
# Éditer .env : renseigner JWT_SECRET (obligatoire)

# 3. Démarrer le serveur
npm start
# → http://localhost:3001
# → Le frontend est servi automatiquement depuis /frontend

Production (Railway)

  1. Connecter le dépôt GitHub au service Railway
  2. Build command : (aucun — Node.js)
  3. Start command : cd backend && npm start
  4. Ajouter un volume persistant monté sur /app/backend/database pour conserver zenith.db entre les déploiements
  5. Configurer les variables d'environnement (voir §11)
  6. Railway assigne automatiquement le domaine et injecte PORT
✅
Vérification post-déploiement Accéder à https://votre-domaine.railway.app/api/health — retourne { status: "ok", db_version: 27, tables: [...] }. Si db_version est inférieur à 27, les migrations en attente s'appliquent automatiquement au prochain démarrage.

Sauvegardes automatiques

Avant chaque migration, le système crée une copie zenith.db.v{N}.bak dans le même dossier. Ces fichiers peuvent être supprimés manuellement une fois la migration validée. Les fichiers .v14.bak et .v23.bak visibles dans le dépôt correspondent aux dernières migrations appliquées.
